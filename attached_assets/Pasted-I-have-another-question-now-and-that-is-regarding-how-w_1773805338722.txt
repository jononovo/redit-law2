Good questions. Let me think through both.

On question 1 — folder per merchant vs. single folder with all merchants:

Folder per merchant is the stronger choice. Here's why:

Right now you have platforms/SHOPIFY.md and checkouts/SHOPIFY.md — two files about the same merchant split across two folders organized by phase. That means when you want to update everything about Shopify, you're hunting across two directories. A folder-per-merchant structure flips this: shopify/platform.md and shopify/checkout.md live together. Everything about one merchant in one place.

The metadata file point you raised is the real win. A shopify/meta.json could carry structured data — platform detection signals, agent friendliness score, supported checkout methods, last verified date. The PROCUREMENT.md detection script could eventually be generated from these meta files rather than hand-maintained. And when the Skill Builder auto-generates vendor skills, it already produces a skill.json per vendor — that maps cleanly to a meta.json in a vendor folder.

It also aligns with the standalone skill direction from your earlier conversation — each merchant folder can optionally include a SKILL.md (or INDEX.md) that makes it usable independently if someone only needs that one merchant.

The "agents" folder pattern (single folder, one file per agent) works there because agent files are thin wrappers with minimal cross-referencing. Merchant files are heavier — they have platform-specific detection, navigation patterns, checkout quirks, and potentially multiple files as merchants get more complex. Amazon is already complex enough that it could warrant separate files for search, checkout, and order tracking. A single merchants/ folder with flat files would get crowded fast.

On question 2 — unifying procurement and checkout into one file per merchant:

I'd unify them. For Shopify specifically, platforms/SHOPIFY.md is 142 lines and checkouts/SHOPIFY.md is 153 lines. Combined that's around 295 lines — manageable for a single file. And the content flows naturally: detect → navigate → browse → add to cart → fill checkout form. Splitting that across two files forces the agent to context-switch mid-purchase for no real benefit.

The only reason they were separate was because the old structure organized by phase (platforms/ and checkouts/), not by merchant. Once you organize by merchant, the split doesn't serve a purpose anymore.

For the generic files it's a tighter call — platforms/GENERIC.md covers WooCommerce, Squarespace, BigCommerce, and unknown sites all in one file, and checkouts/GENERIC.md covers Stripe, Braintree, Adyen, inline fields, and multi-step. Those are broader reference files, not merchant-specific. I'd keep generic as two files (navigation and checkout) since they're not really "one merchant" — they're fallback reference guides that cover many scenarios.

So the structure I'd suggest:

shopify/
  SHOPIFY.md          ← unified: detection, navigation, checkout
  meta.json           ← structured metadata

amazon/
  AMAZON.md           ← unified: detection, navigation, checkout (saved payment methods)
  meta.json

generic/
  NAVIGATION.md       ← browsing patterns for WooCommerce, Squarespace, BigCommerce, unknown
  CHECKOUT.md         ← form filling for Stripe, Braintree, Adyen, inline, multi-step

PROCUREMENT.md stays as the routing brain that points to the right merchant folder. The per-merchant files become self-contained — an agent reads PROCUREMENT.md, gets told "you're on Shopify," and then reads one file (shopify/SHOPIFY.md) that takes it from navigation through checkout.
