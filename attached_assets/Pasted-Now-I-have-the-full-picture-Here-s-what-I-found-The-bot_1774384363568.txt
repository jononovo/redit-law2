Now I have the full picture. Here's what I found:
The bot-facing API (/api/v1/bot/skills) has been switched over to the database. It queries brand_index via storage.searchBrands() and returns results from the DB. That's done.
But the human-facing catalog UI is still reading from the in-memory TypeScript registry. Both app/skills/page.tsx (the catalog) and app/skills/[vendor]/page.tsx (the vendor detail page) import VENDOR_REGISTRY and getVendorBySlug() from lib/procurement-skills/registry.ts. They do all filtering in JavaScript with .filter() chains on the in-memory array.
This means there are two sources of truth running simultaneously: the database for agents, and the TypeScript files for human visitors. If someone claims a brand and the maturity updates to "official" in brand_index, the bot API reflects it immediately, but the catalog UI still shows whatever the TypeScript file says. If a new brand gets added to the database via the skill builder pipeline, it appears in the bot API but is invisible on the catalog page.
That's the most urgent thing to address — more urgent than claims UI polish.
Beyond that, here are the other items I spotted:
The export route (/api/v1/skills/export) still imports VENDOR_REGISTRY from the TypeScript registry. When exporting to ClawHub or skills.sh, it's reading from the old source. This should be switched to read from brand_index too.
