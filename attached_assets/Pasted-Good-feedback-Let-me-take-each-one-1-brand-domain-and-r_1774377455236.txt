Good feedback. Let me take each one:
1. brand_domain and role on brand_claims — agree, add both. The domain extraction needs to be stored for audit trail (you want to see why a claim was auto-verified months later). And the admin/editor role distinction is real — the first person from @staples.com to claim gets admin, the second gets editor. Without the role column, you'd have to infer it from insertion order, which is fragile. Two columns, clear value.
2. FK ordering dependency — noted but already handled. They acknowledged the phases cover this. No action needed, but worth the implementing agent knowing: brand_claims migration runs in Phase 3, after Phase 1 seeds brand_index.
3. category backward compatibility — this is the one that matters most. I'd accept category as a query parameter alias for sector in the API, mapping it internally. The API handler just does if (params.category) params.sector = params.category. No extra column, no ambiguity in the database, and existing agents don't break. Document that category is deprecated in favor of sector and will be removed in a future version. The implementing agent needs to know this explicitly.
4. checkout_methods placement at top level — correct, they're confirming the right call. Checkout methods describe CreditClaw's execution path, not the brand's buying configuration. Top level alongside capabilities is right. No change needed.
5. Manual review queue for unmatched claims — they're right to flag it, but it's not Phase 3 scope. Unverified claims sit as status: 'pending' rows in brand_claims. For Phase 3, that's sufficient — an admin can query SELECT * FROM brand_claims WHERE status = 'pending' directly. A proper admin UI and review endpoints are a Phase 5 concern. The plan should note this as a future requirement so pending claims don't get forgotten, but it shouldn't block the build.
