In Next.js (especially App Router), there isn't one official blessed structure, but the community has converged on a few patterns. Here's how it actually shakes out.
The app/ vs lib/ vs server/ split
app/ is for routing only — pages, layouts, route handlers, loading/error UI. Think of it as the "URL → UI" mapping. You don't put business logic here; you put the thin glue that wires a route to your features.
lib/ is the catch-all that everyone abuses. Strictly speaking it should be for shared, generic utilities — a date formatter, a fetch wrapper, a logger, your Prisma client instance. Stuff that isn't tied to any one feature.
server/ isn't a Next.js convention per se, but people use it (or lib/server/) to mark code that must never ship to the client — DB queries, secret handling, server-only services. It's a safety boundary as much as an organizational one.
The reason your project looks like chaos is that everything got dumped into lib/ regardless of whether it's a generic utility or a whole feature. That's the actual antipattern.
Where features should live: the features/ (or modules/) folder
The pattern most serious Next.js codebases use now is feature-based modularization, often called "screaming architecture" — when you open the repo, the folder names scream what the app does, not what framework it uses.

Yes — features have sub-features
This is exactly the modularization you're asking about. A feature like rail5 is big enough that it has its own internal structure: encryption is one concern, sub-agent lifecycle is another, guardrail evaluation is another. Each becomes a sub-folder inside features/rail5/. From the outside, the rest of the app only imports from features/rail5 (via its index.ts), and doesn't reach into the guts. That's the encapsulation win — same principle as a Python package or a Java module.
The rule of thumb: if two pieces of code only ever touch each other and nothing else, they belong in the same feature folder. If a utility is used by three or more unrelated features, then it graduates to lib/.
Practical rules that keep it sane
Keep app/ files tiny — ideally a route file is 10–30 lines that imports a server action or a component from features/ and renders it. If your page.tsx is 400 lines, the logic belongs in a feature.
Use an index.ts barrel in each feature to define its public surface. Anything not exported from there is private to the feature. This is the closest JS gets to "private" and it works if the team respects it.
Co-locate aggressively. The component, its hook, its server action, its zod schema, and its tests all live next to each other in the feature folder. Resist the urge to make top-level hooks/, types/, schemas/ folders — that's the old "group by file type" pattern and it's what got your lib/ into its current state.
Mark server-only code explicitly. Either put it under features/x/server/ or use the server-only package at the top of those files so importing them from a client component throws at build time. Given Rail5 handles decryption keys, this matters a lot for you.

The heavier touch is to rename lib/ to features/ (or modules/) and move only the genuinely generic utilities — a db.ts, a logger.ts, a utils.ts — into a smaller lib/.
