I found three issues in the plan that need correction before building. None are blockers, but they'd cause problems if I built as written:

Issue 1: The signal function refactor is unnecessarily risky

The plan says to change all 10 signal functions from scoreJsonLd(html) to scoreJsonLd(input: ScoreInput). That touches 10 functions across 3 files plus the compute entry point. Every one of those edits is a chance to introduce a bug in the existing scoring logic.

Safer approach: Keep all existing signal functions exactly as they are. Add a new step in compute.ts — call it boostFromAdditionalPages(signals, additionalPages) — that checks the extra pages and applies bonus points using the same floor principle. This follows the exact same pattern as enhanceScores() from Claude. Zero risk to existing scoring.

Issue 2: Fetching is not fully parallel — it's two waves

The plan says "all fetched in parallel" but that's wrong. We need the sitemap content to discover product URLs. So it's actually:

Wave 1: homepage + sitemap + robots (parallel, current)
Wave 2: product pages + cart + search (parallel, but only after Wave 1 finishes so we can parse the sitemap)
This adds ~3-6 seconds. Total scan time becomes ~8-15 seconds instead of the "5-12 seconds" I estimated. Still acceptable for a one-time deep scan, but I should be honest about it.

Issue 3: Firecrawl for product pages is wasteful

The plan says to use Firecrawl for product pages. But e-commerce sites almost always server-render product pages because they need Google to crawl and index them for SEO. The JS-rendering problem is primarily a homepage issue (SPA shells). Using raw fetch for product pages saves Firecrawl credits (500/month free tier ÷ 3 calls/scan = only 166 scans/month vs 500 scans/month with 1 call).

Correction: Firecrawl for homepage only (current behavior). Raw fetch for product, cart, and search pages. This keeps the rate limit at 500 scans/month and product pages will still have good content.
