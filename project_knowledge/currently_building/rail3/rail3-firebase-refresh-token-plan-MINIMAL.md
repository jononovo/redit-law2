# Rail 3 — Firebase Refresh Token (Minimal, Production-Safe)

## Why this exists

Crossmint's agentic-commerce write endpoints (`/order-intents/:id/credentials`, etc.) require a Firebase ID token as Bearer. The owner's browser has one; a headless bot does not.

We need: keep enough Firebase auth per owner that the BFF can mint a fresh ID token on demand, with no live user.

## What we're storing — and why encryption is the floor

A Firebase **refresh token** is essentially permanent (until password change / admin revoke / Google revoke). Anyone holding one can mint ID tokens as that user. That's the same operational sensitivity as a password. Plaintext in a DB column would mean any backup leak, any DB dump, any read-only credential exposure = full session compromise for every owner, including the ability to operate their Crossmint payment methods.

For a financial product this is non-negotiable. AES-256-GCM with one env-var key is the floor.

What we are **not** doing in this pass, with rationale:
- **No dedicated table.** A column on `users` is fine. Add a table later only if access patterns demand it.
- **No reauth UI banner.** Return 412 `reauth_required` from the bot endpoint; surface a one-line note on the bot card. Fancier UX later.
- **No audit log of token use.** Fine. Add later if compliance asks.
- **No key rotation tooling.** Manual re-encrypt-in-place migration if ever needed.

Everything else is included because it's required for the happy path to work at all.

## Auth model (committed)

```
bot  ──(bot-token)──▶  our BFF  ──(client-key + fresh JWT)──▶  Crossmint
                          ▲
                          │ exchange refresh_token → ID token (cached ~55 min)
                          ▼
                   Google Identity Toolkit
                   (securetoken.googleapis.com)
```

- One Firebase refresh token per owner, AES-256-GCM encrypted at rest.
- BFF maintains an in-process LRU of fresh ID tokens (~55 min TTL).
- On Google 401 during exchange: clear the row's stored token, return 412 from the calling route.
- Bot never sees refresh token, ID token, or Crossmint client key.

## Step 1 — Storage

**Schema change** (`shared/schema.ts`):

Add to the existing `users` table:
```ts
firebaseRefreshTokenCiphertext: text("firebase_refresh_token_ciphertext"),
firebaseRefreshTokenUpdatedAt: timestamp("firebase_refresh_token_updated_at"),
```

Single column nullable so existing rows don't need a backfill. Drizzle `db:push`.

**Storage fragment update** (`server/storage/users.ts` or wherever owner reads/writes live):
- `setFirebaseRefreshToken(ownerUid, ciphertext): Promise<void>`
- `getFirebaseRefreshToken(ownerUid): Promise<string | null>` (returns ciphertext, not plaintext — caller decrypts)
- `clearFirebaseRefreshToken(ownerUid): Promise<void>`

## Step 2 — Encryption helper

**New file:** `features/platform-management/auth/firebase-refresh-crypto.ts`

- AES-256-GCM via Node's built-in `crypto`.
- Key from `OWNER_REFRESH_TOKEN_ENCRYPTION_KEY` env (32 raw bytes, base64-encoded). Helper throws explicit on missing key (no silent fallback per house rule).
- `encryptRefreshToken(plain: string): string` returns `base64(iv).base64(ciphertext).base64(authTag)` (dot-separated for trivial parsing).
- `decryptRefreshToken(stored: string): string` inverse.
- ~30 lines total.

**Secret to request from user via secrets flow:** `OWNER_REFRESH_TOKEN_ENCRYPTION_KEY` — generated by user via `openssl rand -base64 32` (or we generate and they paste). This is **our** key, not anything Firebase issues.

## Step 3 — Capture during pairing

**New endpoint:** `POST /api/v1/auth/firebase-refresh-token`
- Session-cookie protected (only the signed-in owner can write their own token).
- Body: `{ refresh_token: string }`.
- Encrypts → upserts via storage helper → returns `{ ok: true }`.

**Client change:** during the bot pairing flow (existing component under `features/platform-management/bot-management/`), after Firebase sign-in is confirmed and before the bot is marked active, call this endpoint once with `currentUser.refreshToken`.

Idempotent. Re-pairing overwrites.

## Step 4 — Token exchange helper

**New file:** `features/platform-management/auth/firebase-token-exchange.ts`

```ts
const cache = new Map<string, { idToken: string; expiresAt: number }>();

export async function getFreshIdToken(ownerUid: string): Promise<string> {
  const cached = cache.get(ownerUid);
  if (cached && cached.expiresAt > Date.now() + 60_000) return cached.idToken;

  const ciphertext = await storage.getFirebaseRefreshToken(ownerUid);
  if (!ciphertext) throw new ReauthRequiredError(ownerUid, "no_refresh_token");

  const refreshToken = decryptRefreshToken(ciphertext);
  const res = await fetch(`https://securetoken.googleapis.com/v1/token?key=${FIREBASE_WEB_API_KEY}`, {
    method: "POST",
    headers: { "Content-Type": "application/x-www-form-urlencoded" },
    body: `grant_type=refresh_token&refresh_token=${encodeURIComponent(refreshToken)}`,
  });
  if (!res.ok) {
    await storage.clearFirebaseRefreshToken(ownerUid);
    throw new ReauthRequiredError(ownerUid, `google_${res.status}`);
  }
  const { id_token, refresh_token: rotated, expires_in } = await res.json();

  // Google occasionally rotates the refresh token. Upsert if changed.
  if (rotated && rotated !== refreshToken) {
    await storage.setFirebaseRefreshToken(ownerUid, encryptRefreshToken(rotated));
  }

  const idToken = id_token as string;
  const expiresAt = Date.now() + (Number(expires_in) - 60) * 1000;
  cache.set(ownerUid, { idToken, expiresAt });
  return idToken;
}
```

- `FIREBASE_WEB_API_KEY` — same value already in the client-side Firebase config (the "API key" from the web app config). Not a secret in the traditional sense (it's exposed to every browser), but needs to be available server-side. If it's not already in the env, request it via the secrets flow.
- `ReauthRequiredError` extends `Error` with `ownerUid` field; thrown to callers.

## Step 6 — Backfill posture

No automatic backfill. Owners with bots paired before this lands have no refresh token on file. Their bot purchases return 412 until they revisit the pairing UI once. We surface a single line on the bot card: "Re-pair to enable autonomous purchases." No banner, no nag.

## Acceptance

- Bot `POST /api/v1/bot/rail3/checkout` succeeds against staging Crossmint without a live browser session.
- `/order-intents/:id/credentials` returns 200 (was 403).
- Refresh token in DB is opaque (`AAA...:BBB...:CCC...` format), never plaintext.
- Cache: second bot purchase within an hour does NOT hit `securetoken.googleapis.com`.
- Bad refresh token → 412 from the bot endpoint, row cleared, owner sees re-pair prompt.

## Out of scope (don't build)

- Dedicated `owner_firebase_credentials` table.
- Multi-device refresh tokens.
- Reauth UI banner / modal.
- Audit log of token exchanges.
- Key rotation automation.
- Backfill for existing owners.
- Crossmint master-agent (shared-agentId) pattern — different design, decoupled from this work.

## Risks acknowledged

- We hold permanent-ish credentials per user. Encrypted at rest, never logged. Standard secrets hygiene applies.
- A leaked `OWNER_REFRESH_TOKEN_ENCRYPTION_KEY` + DB dump = total compromise. Same shape as any other AES-at-rest scheme. Mitigation: keep the key in Replit secrets, never in the codebase.
- Google token-exchange rate limits: generous; the cache keeps us well under.
