# Skill Variants — Build and Deployment Guide

## Overview

This system generates variant OpenClaw skill packages from the master files in `public/`. Each variant has its own config, and all variants share the same file set with differences in name, description, version, URL prefix, and title.

---

## How the Build Works

### Master Files

All skill content lives in `public/` — this is your single source of truth. Edit files here, and all variants pick up the changes on the next build.

### Variant Configs

Each variant has an independent config at `skill-variants/<name>/variant.config.json`. To change a single variant, edit its config. No central manifest exists — each variant is fully self-contained.

Example config:

```json
{
  "source": "public",
  "urlPrefix": "stripe",
  "overrides": {
    "name": "creditclaw-stripe",
    "version": "2.3.1",
    "description": "Stripe-focused skill description"
  },
  "skillJsonOverrides": {
    "name": "creditclaw-stripe",
    "version": "2.3.1",
    "description": "Stripe-focused skill description"
  },
  "titleOverride": "CreditClaw Stripe — Payments for AI Agents"
}
```

### Running a Build

```bash
npx tsx skill-variants/build-variants.ts
```

This reads every `skill-variants/<name>/variant.config.json`, processes the master files, and writes output to `skill-variants/<name>/dist/`.

### What the Build Does

1. Copies all `.md` and `.json` files from `public/`
2. Patches YAML frontmatter with config overrides
3. Rewrites all `creditclaw.com/<file>` URLs to include the variant's `urlPrefix`
4. Replaces the H1 title if `titleOverride` is set
5. Auto-generates the Skill Files table and curl install commands
6. Generates a patched `skill.json` with correct file URLs
7. Copies any `extraFiles` listed in the config

### Adding a New Variant

1. Create a folder: `skill-variants/my-variant/`
2. Add a `variant.config.json` with your overrides
3. Run the build
4. Output appears in `skill-variants/my-variant/dist/`

## How ClawHub Publishing Works

Each `clawhub publish` call requires these flags:

| Flag | Purpose | Where it comes from |
|------|---------|-------------------|
| `--slug` | Unique identifier on ClawHub (URL-safe, lowercase) | `name` field in `skill.md` frontmatter or `variant.config.json` |
| `--version` | Semver version for this publish | `version` field in `skill.md` frontmatter or `variant.config.json` |
| `--tags latest` | Points the `latest` tag to this version | Always passed |
| `--changelog` | Description of what changed | Auto-generated by workflow |

Each publish is a **full replacement** — ClawHub creates a new version with all files from the folder. There are no partial updates. If you skip a field, it won't carry over from the previous version.

### Slug Rules

- Must be lowercase and URL-safe: `^[a-z0-9][a-z0-9-]*$`
- Main skill slug: `creditclaw` (from `name: creditclaw` in `public/skill.md`)
- Variant slugs: `creditclaw-stripe`, `creditclaw-creditcard`, etc. (from `overrides.name` in each variant config)

### Version Bumping

Every publish to ClawHub must have a **new version number**. If you try to publish the same version twice, it will fail. When you update skill content:

1. Bump the `version` field in `public/skill.md` frontmatter
2. For variants, bump the `version` in each `variant.config.json` (both `overrides.version` and `skillJsonOverrides.version`)

## Deployment to ClawHub

### Current Status

The deployment pipeline is set up in two phases:

- **Phase 1 (active now):** Only the main skill in `public/` is auto-published to ClawHub
- **Phase 2 (when ready):** Variant publishing is added by uncommenting lines in the workflow file

This lets you verify the main skill publishes and updates correctly before turning on variants.

### Automated Publishing via GitHub Actions

Publishing to ClawHub is handled automatically through GitHub Actions. When you push changes to `main` that affect skill files, GitHub detects the changes and runs the publish pipeline.

#### What Triggers a Publish (Phase 1 — Main Skill Only)

The workflow currently runs when these files change on `main`:

- `public/*.md` — master skill markdown files
- `public/*.json` — master skill.json

Normal app code changes (frontend, backend, styles, etc.) do NOT trigger a publish. Your Replit app deployment and ClawHub skill publishing are completely independent.

#### What the Pipeline Does (Phase 1)

1. **Extracts metadata** — reads `name` and `version` from `public/skill.md` frontmatter
2. **Publishes the main skill** — runs:
   ```bash
   clawhub publish ./public \
     --slug creditclaw \
     --version 2.5.0 \
     --tags latest \
     --no-input
   ```

That's it for now. Variant publishing is commented out in the workflow file.

#### Workflow File

The workflow reference copy lives at `skill-variants/publish-skills.yml`. To activate it on GitHub,
copy this file to `.github/workflows/publish-skills.yml` in your GitHub repository (via the
GitHub web editor or a local clone). This is a one-time setup step — see "Setup (One-Time)" below.

The `.github/` folder is not managed by Replit to avoid sync conflicts.

## Enabling Variant Publishing (Phase 2)

When you're confident the main skill is publishing and updating correctly on ClawHub, enable variant auto-publishing by making these changes in `skill-variants/publish-skills.yml` (and re-copying it to GitHub):

### Step 1: Uncomment the path triggers

Find these commented lines near the top of the file (around lines 9-10):

```yaml
      # UNCOMMENT THESE when ready to auto-publish variants (see DEPLOYMENT.md "Enabling Variant Publishing")
      # - 'skill-variants/**/variant.config.json'
      # - 'skill-variants/build-variants.ts'
```

Change them to:

```yaml
      - 'skill-variants/**/variant.config.json'
      - 'skill-variants/build-variants.ts'
```

This tells GitHub to also trigger the workflow when variant configs or the build script change.

### Step 2: Uncomment the publish-variants job

Find the commented `publish-variants` job block (around lines 45-77). Remove the `#` from every line in that block, so it goes from:

```yaml
  # publish-variants:
  #   name: Publish Skill Variants
  #   runs-on: ubuntu-latest
  #   needs: publish-main-skill
  #   steps:
  #     ...
```

To:

```yaml
  publish-variants:
    name: Publish Skill Variants
    runs-on: ubuntu-latest
    needs: publish-main-skill
    steps:
      ...
```

### Step 3: Push to main

Commit and push. The next time any skill file changes, both the main skill and all variants will auto-publish.

### What Changes in Phase 2

The full pipeline becomes:

1. **Publishes the main skill** — with `--slug`, `--version`, and `--tags latest` extracted from `public/skill.md`
2. **Builds all variants** — runs `npx tsx skill-variants/build-variants.ts` to generate each variant's `dist/` folder
3. **Publishes each variant** — reads `overrides.name` and `overrides.version` from each variant's `variant.config.json`, then runs:
   ```bash
   clawhub publish ./skill-variants/<name>/dist \
     --slug creditclaw-<name> \
     --version <version> \
     --tags latest \
     --no-input
   ```

Variants only publish after the main skill succeeds. If the main skill publish fails, variants are skipped.

## Setup (One-Time)

To enable automated publishing:

1. **Get your ClawHub CLI token**
   - Log into clawhub.ai
   - Go to your account settings
   - Copy your CLI/API token

2. **Add the token to GitHub**
   - In your GitHub repo, go to Settings → Secrets and variables → Actions
   - Click "New repository secret"
   - Name: `CLAWHUB_TOKEN`
   - Value: paste your CLI token

3. **Copy the workflow file to GitHub**
   - The workflow reference copy is at `skill-variants/publish-skills.yml` in this repo
   - On GitHub, create `.github/workflows/publish-skills.yml` and paste the contents
   - You can do this via the GitHub web editor: navigate to your repo → Add file → Create new file → enter the path `.github/workflows/publish-skills.yml` → paste → commit to `main`
   - Once it's there, GitHub starts watching for skill file changes automatically
   - **Note:** The `.github/` folder is managed on GitHub only — not through Replit — to avoid sync conflicts

## Manual Publishing

You can always publish manually from your local machine or from Replit:

```bash
# Install the CLI
npm install -g clawhub

# Authenticate
clawhub login

# Publish the main skill
clawhub publish ./public \
  --slug creditclaw \
  --version 2.5.0 \
  --tags latest

# Build and publish variants (when ready)
npx tsx skill-variants/build-variants.ts

clawhub publish ./skill-variants/stripe/dist \
  --slug stripe --version 2.3.1 --tags latest

clawhub publish ./skill-variants/creditcard/dist \
  --slug creditcard --version 2.3.1 --tags latest
```

### First-Time Publish

Before the GitHub Action can publish automatically, you should do one manual publish to register your skill on ClawHub:

```bash
clawhub login
clawhub publish ./public \
  --slug creditclaw \
  --version 2.5.0 \
  --tags latest
```

This establishes the skill under your account. After that, the GitHub Action handles everything using the `CLAWHUB_TOKEN` secret.

## Summary

- Each publish is a full replacement — all fields and files must be included every time
- Slugs are passed explicitly via `--slug` — folder names don't matter
- Version must be bumped for every publish (ClawHub rejects duplicate versions)
- Edit master files in `public/` → all variants update on next build
- Edit a single variant's config → only that variant changes
- **Phase 1 (now):** Push to `main` with changes in `public/` → auto-publishes main skill only
- **Phase 2 (later):** Uncomment variant lines in workflow → auto-publishes main skill + all variants
- Push to `main` with app-only changes → nothing happens on ClawHub
