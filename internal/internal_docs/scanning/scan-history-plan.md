# Plan: Scan History

**Status:** Not built. Schema designed, implementation pending.

Simple append-only log of every scan. Tracks score changes over time and preserves previous SKILL.md / skill.json content for reference.

---

## Why It Exists

When the scanner re-scans a domain, it overwrites the `brand_index` row. The previous score, SKILL.md, and skill.json are gone. We need a log so we can:

1. Show "Nike's score went from 36 → 58 → 71 over 3 scans"
2. See what the old SKILL.md looked like before a re-scan changed it
3. Know when each scan happened and what tier it was

That's it. No rollback, no diffing, no delta computation. Just a log.

---

## Data Model

### New table: `scan_history`

One row per scan per brand. Append-only — rows are inserted, never updated or deleted.

```sql
CREATE TABLE scan_history (
  id              SERIAL PRIMARY KEY,
  brand_id        INTEGER NOT NULL REFERENCES brand_index(id) ON DELETE CASCADE,
  brand_slug      TEXT NOT NULL,

  -- Score at time of scan
  score           INTEGER,
  score_breakdown JSONB,

  -- Skill files at time of scan
  skill_md        TEXT,
  skill_json      JSONB,

  -- Scan metadata
  scan_tier       TEXT NOT NULL DEFAULT 'free',
  model_used      TEXT,
  maturity        TEXT,

  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX scan_history_brand_id_idx ON scan_history (brand_id, created_at DESC);
CREATE INDEX scan_history_slug_idx ON scan_history (brand_slug, created_at DESC);
```

### Column rationale

| Column | Why |
|---|---|
| `brand_id` + `brand_slug` | FK for joins + slug for human-readable queries. Both because slug is stable and readable, but brand_id is the real FK. |
| `score` + `score_breakdown` | Denormalized from `brand_index` at scan time. `score` is the integer for fast trending queries. `score_breakdown` is the full JSONB for detail views. |
| `skill_md` | The SKILL.md text that was generated by this scan. Can be NULL if the scan didn't produce one. |
| `skill_json` | The skill.json object generated by `buildSkillJson()` at scan time. Stored as JSONB so it's queryable. Can be NULL. |
| `scan_tier` | `'free'` or `'premium'` — distinguishes scan types when premium lands. |
| `model_used` | Which LLM model ran the scan (e.g. `'claude-3-5-haiku-20241022'`). Useful for cost tracking and debugging quality differences. |
| `maturity` | The brand's maturity level at the time of the scan (`draft`, `beta`, `verified`, etc.). |

### What's NOT in this table

- No `brand_snapshot` JSONB — we don't need a full copy of the brand_index row
- No `score_delta` — compute it at query time if needed (`score - LAG(score)`)
- No `changed_fields` — not tracking what changed between scans
- No `version_number` — `created_at` ordering is sufficient
- No `content_hash` — not deduplicating files

## Write Path

The scan already follows this flow:

```
1. Compute score (computeScoreFromRubric)
2. Build VendorSkill draft (buildVendorSkillDraft)
3. Generate SKILL.md (generateVendorSkill)
4. Upsert brand_index (storage.upsertBrandIndex)
```

Add one step after the upsert:

```
5. INSERT into scan_history with the new score, SKILL.md, and skill.json
```

We log the NEW state (what the scan just produced), not the old state being overwritten. This is simpler — the scan already has all the new data in memory.

### Files to modify

| File | Change |
|---|---|
| `shared/schema.ts` | Add `scanHistory` table definition |
| `server/storage.ts` | Add `insertScanHistory()` method |
| `app/api/v1/scan/route.ts` | Call `insertScanHistory()` after upsert (lines ~200-240) |
| `lib/scan-queue/process-next.ts` | Same — call `insertScanHistory()` after upsert (lines ~146-190) |

### The insert call

```typescript
await storage.insertScanHistory({
  brandId: upserted.id,
  brandSlug: slug,
  score: scoreResult.overallScore,
  scoreBreakdown: scoreResult.breakdown,
  skillMd: skillMd,
  skillJson: buildSkillJson(/* brand record */),
  scanTier: scanTier ?? "free",
  modelUsed: "claude-3-5-haiku-20241022",
  maturity: upserted.maturity,
});
```

This is fire-and-forget — if the insert fails, the scan still succeeds. Wrap in try/catch, log the error, move on.

## Read Path

### Score history for a brand

```sql
SELECT score, score_breakdown, scan_tier, created_at
FROM scan_history
WHERE brand_slug = $1
ORDER BY created_at ASC;
```

### API endpoint

`GET /api/v1/brands/{slug}/history`

Response:
```json
{
  "slug": "nike",
  "current_score": 71,
  "scans": [
    { "score": 36, "scan_tier": "free", "maturity": "draft", "created_at": "2026-03-28T12:00:00Z" },
    { "score": 71, "scan_tier": "free", "maturity": "verified", "created_at": "2026-04-05T12:00:00Z" }
  ]
}
```

No auth required. Cached: 10 min TTL.
