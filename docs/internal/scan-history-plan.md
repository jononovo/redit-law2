# Scan History ‚Äî Technical Plan

*Simple append-only log of every scan. Tracks score changes over time and preserves previous SKILL.md / skill.json content for reference.*

---

## Why This Exists

When the scanner re-scans a domain, it overwrites the `brand_index` row. The previous score, SKILL.md, and skill.json are gone. We need a log so we can:

1. Show "Nike's score went from 36 ‚Üí 58 ‚Üí 71 over 3 scans"
2. See what the old SKILL.md looked like before a re-scan changed it
3. Know when each scan happened and what tier it was

That's it. No rollback, no diffing, no delta computation. Just a log.

---

## Data Model

### New table: `scan_history`

One row per scan per brand. Append-only ‚Äî rows are inserted, never updated or deleted.

```sql
CREATE TABLE scan_history (
  id              SERIAL PRIMARY KEY,
  brand_id        INTEGER NOT NULL REFERENCES brand_index(id) ON DELETE CASCADE,
  brand_slug      TEXT NOT NULL,

  -- Score at time of scan
  score           INTEGER,
  score_breakdown JSONB,

  -- Skill files at time of scan
  skill_md        TEXT,
  skill_json      JSONB,

  -- Scan metadata
  scan_tier       TEXT NOT NULL DEFAULT 'free',
  model_used      TEXT,
  maturity        TEXT,

  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX scan_history_brand_id_idx ON scan_history (brand_id, created_at DESC);
CREATE INDEX scan_history_slug_idx ON scan_history (brand_slug, created_at DESC);
```

### Column rationale

| Column | Why |
|---|---|
| `brand_id` + `brand_slug` | FK for joins + slug for human-readable queries. Both because slug is stable and readable, but brand_id is the real FK. |
| `score` + `score_breakdown` | Denormalized from `brand_index` at scan time. `score` is the integer for fast trending queries. `score_breakdown` is the full JSONB for detail views. |
| `skill_md` | The SKILL.md text that was generated by this scan. Can be NULL if the scan didn't produce one. |
| `skill_json` | The skill.json object generated by `buildSkillJson()` at scan time. Stored as JSONB so it's queryable. Can be NULL. |
| `scan_tier` | `'free'` or `'premium'` ‚Äî distinguishes scan types when premium lands. |
| `model_used` | Which LLM model ran the scan (e.g. `'claude-3-5-haiku-20241022'`). Useful for cost tracking and debugging quality differences. |
| `maturity` | The brand's maturity level at the time of the scan (`draft`, `beta`, `verified`, etc.). |

### What's NOT in this table

- No `brand_snapshot` JSONB ‚Äî we don't need a full copy of the brand_index row
- No `score_delta` ‚Äî compute it at query time if needed (`score - LAG(score)`)
- No `changed_fields` ‚Äî not tracking what changed between scans
- No `version_number` ‚Äî `created_at` ordering is sufficient
- No `content_hash` ‚Äî not deduplicating files

## Write Path

The scan already follows this flow:

```
1. Compute score (computeScoreFromRubric)
2. Build VendorSkill draft (buildVendorSkillDraft)
3. Generate SKILL.md (generateVendorSkill)
4. Upsert brand_index (storage.upsertBrandIndex)
```

Add one step after the upsert:

```
5. INSERT into scan_history with the new score, SKILL.md, and skill.json
```

We log the NEW state (what the scan just produced), not the old state being overwritten. This is simpler ‚Äî the scan already has all the new data in memory.

### Files to modify

| File | Change |
|---|---|
| `shared/schema.ts` | Add `scanHistory` table definition |
| `server/storage.ts` | Add `insertScanHistory()` method |
| `app/api/v1/scan/route.ts` | Call `insertScanHistory()` after upsert (lines ~200-240) |
| `lib/scan-queue/process-next.ts` | Same ‚Äî call `insertScanHistory()` after upsert (lines ~146-190) |

### The insert call

```typescript
await storage.insertScanHistory({
  brandId: upserted.id,
  brandSlug: slug,
  score: scoreResult.overallScore,
  scoreBreakdown: scoreResult.breakdown,
  skillMd: skillMd,
  skillJson: buildSkillJson(/* brand record */),
  scanTier: scanTier ?? "free",
  modelUsed: "claude-3-5-haiku-20241022",
  maturity: upserted.maturity,
});
```

This is fire-and-forget ‚Äî if the insert fails, the scan still succeeds. Wrap in try/catch, log the error, move on.

## Read Path

### Score history for a brand

```sql
SELECT score, score_breakdown, scan_tier, created_at
FROM scan_history
WHERE brand_slug = $1
ORDER BY created_at ASC;
```

Returns a timeline: `[{score: 36, date: "2026-03-28"}, {score: 58, date: "2026-04-01"}, {score: 71, date: "2026-04-05"}]`

### API endpoint

`GET /api/v1/brands/{slug}/history`

Response:
```json
{
  "slug": "nike",
  "current_score": 71,
  "scans": [
    {
      "score": 36,
      "scan_tier": "free",
      "maturity": "draft",
      "created_at": "2026-03-28T12:00:00Z"
    },
    {
      "score": 58,
      "scan_tier": "free",
      "maturity": "beta",
      "created_at": "2026-04-01T12:00:00Z"
    },
    {
      "score": 71,
      "scan_tier": "free",
      "maturity": "verified",
      "created_at": "2026-04-05T12:00:00Z"
    }
  ]
}
```

No auth required. Cached: 10 min TTL. The `skill_md` and `skill_json` fields are excluded from the list response to keep it lightweight ‚Äî they can be fetched individually if needed.

### Previous SKILL.md for a specific scan

`GET /api/v1/brands/{slug}/history/{id}`

Returns full detail for one scan entry including `skill_md` and `skill_json`. This is for reference only ‚Äî "what did the SKILL.md look like after the March 28 scan?"

## Scale

| Brands | Avg scans/brand | Rows | Table size estimate |
|---|---|---|---|
| 100 | 3 | 300 | ~5 MB |
| 1,000 | 5 | 5,000 | ~50 MB |
| 10,000 | 5 | 50,000 | ~500 MB |

The `skill_md` and `skill_json` columns are the heaviest (5-15 KB per row combined). At 50K rows this is manageable. If it grows past that, the text columns can be moved to object storage ‚Äî but that's a long way off.

The two indexes (`brand_id + created_at DESC` and `brand_slug + created_at DESC`) cover all query patterns.

---

## What This Enables

1. **Score trending** on brand detail pages ‚Äî simple line chart showing score over time
2. **Scan count** ‚Äî "this brand has been scanned 4 times"
3. **Historical SKILL.md reference** ‚Äî see how the skill evolved
4. **Scan tier tracking** ‚Äî when premium scans land, you can see free vs premium results side by side
5. **Model cost visibility** ‚Äî `model_used` column shows which LLM was used per scan
